# 1\. Executive Summary

This report presents a comprehensive evaluation of AI-powered tools for the Security & Compliance phase of the AI Development Life Cycle (AI-DLC). Based on the project plan, four key activity areas were assessed: vulnerability scanning (SAST), dependency audits (SCA), compliance checks, and secrets management. Three AI integration points were targeted: AI-powered security scanning, auto-fix suggestions, and compliance checklist automation.

Security in 2026 has been transformed by two simultaneous shifts: AI-generated code now accounts for ~41% of new commits and produces vulnerabilities at 1.7x the rate of human-written code, while AI-native security tools now offer reachability analysis, automated fix PRs, and natural language rule creation that traditional SAST tools could not match. The challenge is no longer detection - it is prioritisation, remediation speed, and avoiding alert fatigue (traditional SAST tools generate 80-90% false positives and 70% of alerts are ignored).

Key findings recommend a layered free-first approach: SonarQube Community Edition (already in the Development phase stack) for general SAST and code quality, Semgrep OSS for fast custom-rule security scanning in CI, Snyk Open Source (free tier) for SCA dependency scanning with auto-fix PRs, GitGuardian for secrets detection (free tier covers small teams), and Trivy for container/IaC scanning. Claude Code provides AI-powered remediation across all categories. Total estimated annual cost for a 10-person team: \$0 on the minimal free-only stack, rising to \$3,000-6,000/yr if Snyk Code SAST is added for AI-powered code scanning, or \$15,000+/yr for enterprise platforms (Veracode/Checkmarx).

## 3.1 Vulnerability Scanning - Static Application Security Testing (SAST)

SAST scans source code for security vulnerabilities without running the application. In 2026, the SAST landscape has split into three tiers: traditional pattern-matching scanners (free, fast, noisy), modern AI/ML-powered scanners (paid, accurate, with auto-fix), and AI-native engines (newest, natural language rules, deepest analysis). Research from 2026 shows traditional SAST tools generate 80-90% false positives, with 70% of alerts ignored - making accuracy and noise reduction the most important selection criteria, not just feature checklists.

The challenge has intensified in 2026: GitGuardian's State of Secrets Sprawl 2026 report found 28.65 million new hardcoded secrets added to public GitHub repositories in 2025 alone (a 34% increase), with AI-assisted commits leaking secrets at 3.2% rate (~2x the baseline). The project plan specifies Git Guardian as the chosen tool - this section validates that choice and details the layered defence.

**Primary Recommendation:** GitGuardian is the project plan's chosen tool and the recommendation is validated by 2026 research. It is the most widely adopted secrets detection platform with 500+ secret type detectors (broader than alternatives' 200-350), incident management UI for security teams, real-time alerts on new commits, and historical scanning of entire git history. The free tier covers teams under 25 developers; paid plans scale per developer for larger teams or advanced features. As of March 2026, GitGuardian's ggshield CLI added native hooks for AI coding assistants (Cursor and Claude Code) that catch secrets before they enter prompts or code completions.

**Layered Defence Approach:** Academic research (NC State 2025) found that no single secrets detection tool catches everything - overlap between tools is only 18-76%. Use GitGuardian as the primary platform-level scanner AND a pre-commit hook to catch secrets locally before they ever reach the remote: ggshield (GitGuardian's CLI) integrates as a pre-commit hook. For teams that prefer pure OSS, use Gitleaks (MIT licensed, fast) or detect-secrets as the pre-commit layer. The cost of running multiple tools is near-zero; the cost of missing a secret in production is significant.

**Workflow:** (1) ggshield/Gitleaks runs as a pre-commit hook - secrets blocked before commit. (2) GitGuardian scans every push to remote in real time - alerts within seconds of a leak. (3) GitGuardian scans entire git history on initial setup to catch existing leaks. (4) When a secret is detected, rotation is mandatory - GitGuardian's incident workflow guides the rotation process. Note: 64% of secrets leaked in 2022 were still active in 2026 - detection without rotation is meaningless.

**Cost of Missing This:** This is the highest-impact security tool in the stack. The downside of a leaked secret (cloud account compromise, data breach, customer trust) vastly exceeds GitGuardian's cost. Even teams on a strict \$0 budget should run Gitleaks as a pre-commit hook - it is free, takes 10 minutes to set up, and prevents the most common security mistake developers make.

## 3.5 AI-Generated Code Security (Cross-cutting)

A new security category has emerged in 2026: securing AI-generated code. Cycode's State of Product Security 2026 report found that 100% of surveyed organisations have AI-generated code in their codebases, while 81% lack visibility into AI usage across the SDLC. Common AI code issues include exposed API keys in client-side bundles, missing input validation on generated forms, insecure default configurations, overly permissive CORS headers, and OWASP Top 10 LLM risks (prompt injection, insecure output handling).

**Recommendation:** The AI-DLC's existing AI integration (Cursor + Claude Code) requires no additional AI-code-specific security tool because: (1) CodeRabbit (Phase 3 stack) already reviews AI-generated PRs with AI code awareness, (2) Semgrep + Snyk Code rules now include OWASP LLM Top 10 coverage, (3) GitGuardian's ggshield hooks for Cursor and Claude (March 2026) intercept secrets at the AI prompt level. The key process change is treating AI-generated code with extra scrutiny - this is documented in the Phase 3 Code Review Checklist. No new tools are required.

**Optional: Cycode AI Governance:** For organisations that need formal AI usage visibility (compliance requirement, board mandate), Cycode offers an AI Bill of Materials (AIBOM) and MCP enforcement to control shadow AI across the SDLC. This is enterprise-tier and not needed for typical AI-DLC adoption.

## 7.1 Recommended Security & Compliance Stack

The recommended Security & Compliance stack for the AI-DLC is the free-first layered approach: SonarQube CE + Semgrep OSS for SAST, Dependabot/Renovate + Trivy for SCA, Trivy + Checkov + OPA for IaC compliance, GitGuardian + ggshield for secrets, and Claude Code for AI-powered remediation across all categories. This stack costs \$0 in net-new licensing for a 10-person team.

The stack scales upward when needed: add Snyk Code (~\$3,000/yr) when AI fix automation becomes valuable, add a compliance platform (Vanta/Drata: ~\$7,500-10,000/yr) when formal certification is required, and migrate to enterprise tools (Veracode/Checkmarx) only when working in heavily regulated industries (BFSI, healthcare, government) where the compliance reporting depth is the deciding factor.

## 7.2 Immediate Actions (Week 4)

1\. Add SonarQube CE security gate to CI - SonarQube is already in the Development phase stack for code quality; enable the security rule profile and configure quality gates to block merges with Critical/High vulnerabilities.

2\. Add Semgrep to CI as a separate scan job - enable the OWASP Top 10 ruleset and the language-specific rules for the project's stack. Schedule full repository scans nightly and PR-diff scans on every pull request.

3\. Enable Dependabot for all GitHub repositories (or Renovate for non-GitHub teams). Configure auto-merge for patch-level vulnerability fixes; require PR review for minor/major updates.

4\. Install Trivy in the container build pipeline - fail builds on Critical CVEs in base images or installed packages. Also scan IaC configs (Terraform/Kubernetes) on every PR.

5\. Set up GitGuardian on all repositories AND install ggshield as a pre-commit hook on every developer machine. Run a one-time historical scan to catch any pre-existing leaks.

## 7.3 Process Integration (Week 4)

6\. Define security quality gates (in addition to existing code quality gates from Phase 3): 0 Critical SAST findings on main branch, 0 secrets in code, 0 Critical CVEs in production dependencies (high CVEs require documented exception), all containers scanned and approved before deployment.

7\. Build the Security & Compliance prompt template library: SAST finding remediation prompts, dependency upgrade impact analysis, compliance checklist generation per framework (GDPR, HIPAA, SOC 2), security review prompts for architecture changes.

8\. Document the security incident response workflow: secret leaked → rotate credential immediately → GitGuardian incident workflow → git history scrub if required for compliance → post-mortem. Critical: rotation is mandatory; history scrubbing alone does not remove the risk.

9\. If the project requires compliance certification, evaluate Vanta vs Drata vs Secureframe based on target frameworks (SOC 2 only → Vanta; multi-framework → Secureframe; enterprise/FedRAMP → Drata). Allow 4-6 months for first SOC 2 Type II audit.

## 7.5 Risk Mitigation

The primary risk in security tooling is alert fatigue - traditional SAST tools generate 80-90% false positives, leading to 70% of alerts being ignored. Mitigation: use reachability-aware tools (Snyk, Endor Labs) when budget allows, tune Semgrep rules to the project's specific stack to reduce noise, and route findings to the developer who owns the affected code (not a central security backlog). Track signal-to-noise ratio per tool monthly and disable rules that consistently produce false positives.

A secondary risk is AI-generated code shipping vulnerabilities at 2x the rate of human code. Mitigation is the layered review approach already in the AI-DLC: CodeRabbit catches issues at PR time, SAST runs in CI, ggshield catches secrets at commit time, and Claude Code generates fixes when issues are found. The key is enforcement - quality gates must actually block merges, not just generate warnings.

A third risk is over-investing in compliance platforms before they are needed. Vanta/Drata/Secureframe are useful only when formal certification is the goal. For internal projects without certification requirements, a checklist generated by Claude and tracked in the project's documentation is sufficient. Premature investment in \$10K/year platforms creates ongoing operational overhead without clear value.

A fourth risk is treating secrets detection as 'one tool is enough.' Academic research shows tool overlap is only 18-76% - meaning a single tool misses 24-82% of secrets that another tool would catch. Run GitGuardian platform-side AND a pre-commit hook (ggshield or Gitleaks) developer-side. The cost is near-zero; the cost of a missed leak is significant.
